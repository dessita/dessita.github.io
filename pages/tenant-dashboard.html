Overview

A B2B Web Application where companies can request information to facilitate government financial aid application for programs such as medicaid or to set up a payment plan.

Short Description of the Task

Provide screening service with a Florida Medicaid Eligibility Screening product as a Tenant initiated process within Dashboard.

User Story Epic

` in
201 Dashboard Operators shall be able to view a list of screening products available within the tenant (Florida Medicaid Eligibility will be the first and only screening product for MVP). Post MVP
202 Dashboard Operators shall be able to initiate screening products available within the tenant. (Florida Medicaid Eligibility will be 'Hard-coded' in the platform for MVP. MVP
202.1 Use-case 1: Initiate a screening product for a New consumer MVP
202.1.1 Dashboard shall develop workflows to create new consumers using current state capabilities (Create consumer API's) MVP
202.2 Use-case 2: Initiate a screening product for an Existing consumer MVP
203 Dashboard Operators shall be able to engage with a screening product (Defined as interacting with a series of questions and answers). MVP
203.1 Dashboard shall render the questions, answer options and inputs required based off of data schemas provided by the API. MVP
203.2 Dashboard shall show on-going summary of questions and answers provided during the screening session. MVP
203.3 Dashboard shall show a visual representation of how close to complete the screening progress is based on data provided. MVP
203.4 Dashboard shall show results of screening based on data provided.MVP
203.4.1 Results shall include: Pass / Fail evaluation, Documents that are needed. MVP
204 Positive Path screening evaluation results shall present the Operator with several post evaluation options: MVP
204.1 Use-case: Brand New Consumer (No Doc-order requests) OR Existing Consumer withOUT a doc-order request. MVP
204.1.1 Request Documents Needed: Dashboard will create a new doc-request order for the consumer and automatically seed request with the documents (upload documents) needed (Documents needed shall be a part of the result provided by the API). MVP
204.1.2 Operator shall be prompted if additional requests (Ex. E-sign or Other documents) are desired to be added to the request before finalizing the doc-request order and notifying the consumer. MVP
204.2 Use-case: Existing consumer WITH a pre-existing doc-request order (With previously requested upload documents) MVP
204.2.1 Request Additional Documents Needed: Dashboard will automatically add any additional documents to the doc-request order by cross referencing documents that have already been requested. MVP
204.2.2 Operator shall be prompted if additional requests (Ex. E-sign or Other documents) are desired to be added to the existing doc-request order before notifying the consumer again. MVP
205 Dashboard operators shall be able to print evaluation results, along with Summary of Questions and Answers. MVP
206 Dashboard shall apply/update captured doc-request order ref. numbers to saved screening results. Any screening result, when possible, will be associated with a doc-request order. This will allow our system to track the success of evaluations in a feedback loop. MVP
207 Operators shall be able to review previous screening results of a consumer including when it was done and which operator performed the screening. MVP
208 Operators must confirm to continue the screening process for duplicate screening events.

Ex. If an operator attempts to engage into another screening event for a consumer that already had one done for that same screening product, the operator shall be prompted of the last occurrence of a screening event (and by which operator). The operator must confirm to continue the screening process. 		MVP
209 Operators can access an external link for a screening product. MVP
209.1 Must be accessible after a screening event during the displaying of results. MVP
209.2 Can be accessible whenever the screening product is displayed. MVP
210 Negative Path screening evaluation results shall present the Operator with several post evaluation options: Re-evaluation, Continue with manual requesting of documents. MVP
211 Dashboard shall be capable of allowing a DevOps admin to specify a list of upload document Id's per available Brand. MVP
211.1 Defined Upload list for Brands shall include: PatientClear and FinCertify MVP
211.1.1 PatientClear Upload document Id's: TBD MVP
211.1.2 FinCertify Upload document Id's: TBD MVP
211.2 Upload documents drop-down list shall only be populated with the documents that were included in the defined list for the Brand. MVP
211.2.1 An undefined list for a Brand shall default to populate all available documents retrieved from API. MVP
